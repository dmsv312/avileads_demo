# avileads_demo — просмотр структуры кода

Демо-репозиторий для **ознакомления с организацией кода**. Проект **не предназначен для запуска** — только чтение.

## Как читать репозиторий
- Начните с **`/application`** и **`/services`** — здесь бизнес-логика и сервисы (use-cases).
- Затем **`/repository`** — контракты и реализации доступа к данным.
- **`/http`, `/transport`, `/views`** — обработчики, маршрутизация и шаблоны/статические ресурсы.
- Вспомогательно: **`/models`** (модели/константы), **`/utils`** (утилиты), **`/conf`** (настройки), **`/db`** (структуры/миграции при наличии), `main.go` (точка входа).

## Принципы организации
- **Слои и границы.** Transport → Application → Repository; зависимости направлены внутрь.
- **Сервисы через интерфейсы репозиториев.** Лёгкая подмена хранилища и тестируемость.
- **Явные контракты.** Интерфейсы в репозиториях, простые модели/DTO, без протечки деталей транспорта в доменную логику.
- **Валидация и ошибки — на границах.** Проверки и нормализация входных данных в транспортном слое; бизнес-инварианты — в сервисах.

## Коротко про application-слой
- **Сервисы/use-cases** оформлены как самостоятельные операции.
- **Работают с репозиториями по интерфейсам**, реализации изолированы в `/repository`.
- Общие **сервисы/хелперы** вынесены в `/services` (повторно используемые компоненты).

## Дисклеймеры
- Это **демо**: часть сущностей/фич урезана, данные обезличены, секреты удалены.
- **Запуск не предполагается** (Docker/конфиги приведены для контекстуального чтения).
- Цель — показать подход к **структуре, стилю, слоям и контрактам**, а не полноту продукта.
